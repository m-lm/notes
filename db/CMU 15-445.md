## 01 Relational Model & Algebra

A *database* is NOT a *database management system (DBMS)*. The former refers to the data itself, the latter refers to the software that manages it.

A *data model* is a collection of definitions for what data exists in a database and how they relate.

A *data schema* is a structural description of a particular database, according to a data model.

Relational (SQL) database systems are most popular, and many other data models have been incorporated gradually into mainline SQL systems.

The *relational model* involves: 
1. Structure: defining the database's contents and relations independent of their physical representation (how its stored on disk, etc.)
2. Integrity: ensuring that the database's contents and relations satisfy constraints (such as types, delete behavior, etc.)
3. Manipulation: a declarative, not procedural, API for accessing and updating the database's contents via relations (set theory)

A *relation* is an unordered set containing the relationship of attributes that represent entities (from relational algebra and set theory). A *tuple* is a set of attribute values (*domain*, range of allowed values) in the relation; values are normally atomic/scalar and NULL is a member of every domain.
- *Primary keys* are one or more attributes that can uniquely identify a single tuple so that it can be retrieved. Often, an identity column (like a numerical ID) is used instead of something like a name to ensure uniqueness.
- *Foreign keys* are attributes that point to other relations according a common attribute. This helps model many-to-many relationships between tables. A value has to exist in another relation (e.g. an album ID has to exisst in the artist table).

Thus, an *n*-ary relation is equivalent to a table with *n*-columns. A tuple is like a row.

---

*Data Manipulation Languages (DML)* are APIs for a DBMS that allows users to store and retrieve info.
- *Procedural* languages require the user to specify the strategy to find the desired result. (Relational Algebra)
- *Declarative* requires only that the user specify what data is wanted, and leaves the method for finding it to the DBMS. (SQL, which implements Relational Algebra)

*Relational Algebra* uses fundamental operations that are composed to retrieve and manipulate tuples in a relation. Each operator takes relation(s) as inputs and outputs a new relation.
- Select (WHERE): filtering out tuples based on a predicate (row-wise)
- Projection (SELECT): generates relation containing only specified attributes (column-wise)
- Union, Intersection, Difference
- Product (CROSS JOIN): generates relation that contains all possible combos of tuples from input relations (which do NOT need same attributes; output unions the attributes)
- Join (NATURAL JOIN): generates relation that contains all tuples that are a combination of *n*-tuples from *n* input relations, with common values for one or more attributes

Relation algebra is the backbone for and are declaratively abstracted in SQL-based DBMSs.

---

*Data independence* allows the separation of the user-app level from the low-level data representation. Thus, the best to get the data as well as its storage is abstracted by the DBMS. This also hands off optimization of the physical layout of the data according to environment, contents, and workload to the DBMS.

## 02 Modern SQL

Importantly, SQL is based on unordered *bags* (duplicates) not *sets* (non-duplicates). That's why SQL has keywords like DISTINCT. The underlying relational model itself is based on sets.

GROUP BY basically clusters results according to attribute. Can also have specific group sets. HAVING is like a WHERE clause for GROUP BY result, an extra filter on the aggregration computation.

## 03 Database Storage I

DBMS, at its core, is commonly disk-backed by storing data in non-volatile files on something like SSD, HDD, or network storage. The real interplay to be considered is between the different layers of storage hierarchy: memory (DRAM) and CPU (registers, cache) is fast but scarce, while disk is cheaper but slower. Additionally, it must reckon with databases that exceed the amount of memory available.

The smallest unit DBMSs work with is the *block*, often 4kb or higher.

Because random access is slower than sequential access, we want to optimize read/writes using these blocks and maximize sequential access. Ultimately, we want to reduce writes to random pages, and instead store data in contiguous blocks.
- We may allocate multiple pages simultaneously, known as an *extent*.

Most DBMSs will use multiple files on disk to store and manage all of the data.
- Each DB file on disk can be divided into fixed-length *pages*, which are like blocks. They have headers that store info about what is stored.
- In memory, the *buffer pool* stores *frames* which transfers disk pages of data from disk to memory and vice-versa. It fetches the disk directory that tells where each page is by offset.
- Often, file formats are proprietary, but recently portable file formats like Parquet have become more popular

Database storage is concerned with two problems:
1. How to represent the database in files on disk
2. How to manage memory and move data between disk and memory

The *storage manager* maintains the database's files (reads/writes). It organizes files as a collection of pages while tracking info like available space and read/written data.
- Note: the DBMS does not have multiple copies of a page on disk!
- Formally, a *page* is a fixed-size block of data that can contain tuples, metadata, indexes, log records, etc.
	- Pages types are not mixed, and in some systems pages are self-contained (for example, all useful metadata is inside the page itself).
	- Each page has a *page ID*, which can be unique per DBMS instance, per DB, or even per table.
	- The DBMS has an indirection layer which maps page IDs to physical locations.
- There are 3 types of pages in a DBMS:
	1. Hardware page (4kb) (largest block of data that storage guarantees failsafe writes)
	2. OS page (4kb)
	3. DB page (512b-32kb)
- Size of page depends on workload, environment, and contents:
	- *Read-heavy* workloads tend to larger page size: fetching a single page brings in many tuples needed for a query.
	- *Write-heavy* workloads tend to have smaller page sizes: must write the whole page anyway, even if only some of the data is modified.

Each DBMS manages disk pages in files in different ways:
- Heap File Organization
- Tree File Organization
- Sequential / Sorted (ISAM) File Organization
- Hashing File Organization

A *heap file* is an unordered collection of pages with tuples that are stored in random order. Allows getting, updating, and deleting pages as well as scans, but needs extra metadata for file locations and space availability.

To track location of pages in files (e.g., Table X -> File 1), the DBMS maintains a *page directory* that is synced with the data pages. This is in tandem with tracked page metadata like free space per page, list of free pages, and page types.

*Page headers* are segments within a page containing metadata about its contents, and in some cases is self-contained:
- Page size
- Checksum
- DBMS version
- Transaction visibility
- Compression/encoding metadata
- Schema info
- Data summary/sketches

The question then becomes, once we have database files consisting of pages of headers+data, *how exactly do we organize the data inside of the page?* There are 3 main approaches to organizing the page layout:
1. Tuple-oriented (this course)
2. Log-structured
3. Index-organized

*Slotted pages* is the most common layout scheme which feature a slot array mapping "slots" to tuples' starting offsets.
- Thus, header tracks used slots and the starting offset of the last slot used.

The DBMS assigns each logical tuple a record ID (or row, tuple ID) that represents its physical location in the DB (e.g., file ID>page ID>slot #). Usually, this is derived at runtime as data is read.

Deeper into the page, a *tuple* is just a byte-array prefixed with a metadata header also. The DBMS decodes these bytes into attribute types and values, and its catalog contains schema info about tables to figure out the tuple's layout.

For null attributes, the most common choice in row-stores is a null collumn bitmap header; a bitmap is stored in a centralized header specifying which attributes are null. Column-stores prefer a special placeholder value that represents null.

Most DBMSs do now store tuples exceeding the size of a single page. They store larger values in a separate overflow storage page.

There is also *external value storage*, where large values may not necessarily be appropriate to store in a database, such as a movie. Often they are treated as BLOB types and stored in an external file outside of the DBMS, so it cannot be manipulated (this also means no durability/transaction protections). It is referenced by a pointer in the DB, but not maintained like in a DB.

## 04 Memory Management & Buffer Pools

Database storage is concerned with:
1. Spatial Control (where to write pages on disk): keep pages used together often as close as possible
2. Temporal Control (when to read pages and when to write on update): minimize number of stalls from reading data

The *buffer pool* hold the pages fetched from disk in memory by putting them into *frames*, a byte buffer with offsets to pages. It also has a copy of the page directory.
- When system boots up, call malloc (heap memory allocation) for all the memory you need, not just for the buffer pool.
- The *buffer pool manager* manages the memory that will be utilized by the DBMS.
- OS can handle this, but not always advisable.

The *organization* is laid out as an array of fixed-size pages, which each array entry called a *frame*. When the DBMS requests a page, it places an exact copy into one of these frames.
- Dirty (affected) pages are buffered and not immediately written out (Write-Back Cache, WAL)

Another data structure, the *page table*, tracks pages currently in memory via metadata according to page ID lookups.

A *latch* is basically a lock from OS (a mutex). In DBMSs, a lock protects the DB's logical higher-level contents from other transactions, but latches protect the internal data structures from other workers (e.g. threads).

What is the difference between page table and page directory? It's mainly about disk vs memory:
- Page directory maps _all_ page IDs to page locations in the DB files and recorded on disk.
- Page table maps page IDs to page copies found in buffer pool frames _currently_ in memory.

Why not use the OS (mmap) for managing memory?
- Give up control; the OS does not know about SQL, or DB contents.
- Thus, transactions are not safe since dirty pages can be flushed (written) at any time. This is especially a concern if there is dependence between dirty pages.
- The OS will also stall a thread on page fault because the DBMS does not know which pages are in memory.
- Error handling is more difficult, so page validation suffers.
- Just plain slower; redundant data structure and latches as in DBMS.

So, the DBMS can almost always do a better job than the OS and should control key mechanisms:
- Flushing dirty pages to disk in correct order
- Specialized prefetching
- Buffer replacement
- Thread/process scheduling

The *buffer replacement policy* consists of how the DBMS decides to free frames to make room for new pages. It does so by *evicting* frames from the buffer pool.
- This is a classic problem in computer science; caching.
- One solution is LRU list sorted by the timestamp (metadata) it was last accessed.
	- Can use "Clock" as approximation of LRU: use a reference bit that is set to 1 when a frame's page is accessed since last check, and evicts if not 1.
- LRU + CLOCK are susceptible to *sequential flooding*: a query might read every page in a table but will pollute the buffer pool with pages only needed once.
- Most recently used and least frequently used are also employed.
- *Adaptive replacement policy (ARC)* (2003) is SOTA for buffer replacement algorithms. Has the benefit of LRU and LFU by maintaining two lists and adapts size based on access patterns. Maintains ghost lists to remember evictions.
- Enterprise will try to optimize these policies compared to open source.

*Dirty pages* are modified pages:
- Fast Path: if a page in the buffer pool is not dirty, drop it
- Slow Path: if the page is dirty, then must write back to disk first
- Tradeoff: fast evictions vs. writing dirty pages that won't be used much

*Page cleaning* or *buffer flushing* is when the DBMS goes through the page table and preemptively writes dirty pages to disk; at this point, the page can be evicted or the dirty flag can be reset.

OS/hardware tries to maximize disk bandwidth by reordering/batching I/O requests, but they do not know which requests are most important due to lack of info. The DBMS maintains internal queues to track page I/O requests from entire system, computing priority based on:
- Sequential vs Random I/O
- Critical path task vs Background task
- Table vs Index vs Log vs Ephemeral data
- Transaction info
- User-based SLAs

Most disk ops go through the OS API unless DBMS explicitly avoids. Most DBMSs use *direct I/O* (O_DIRECT) to bypass the OS cache that normally causes redundant page copies, has different eviction policies, and results in loss of control over file I/O.
- Famously, Postgres uses the OS page cache.
- Hardware fsync fails (EIO) can result in the OS telling the dirty pages were written successfully when it wasn't...
